1. System Overview

RigaCap is a serverless, event-driven trading platform built on AWS. The system ingests market data from 4,000+ US equities via a dual-source pipeline (Alpaca SIP primary + yfinance fallback), generates AI-powered trading signals, and delivers them to subscribers via a React dashboard and automated email pipeline. The backend runs on two specialized AWS Lambda functions (API + Worker) from a single container image, using FastAPI + Mangum for ASGI compatibility.

5. Data Pipeline & Caching

Dual-Source Ingestion

Alpaca SIP (primary, Adjustment.SPLIT) + yfinance (fallback). Daily download of 252-day rolling OHLCV for 4,500+ symbols. Auto-failover. Split-adjustment made permanent Apr 2026 to prevent unadjusted-split artifacts (NVDA/GOOG/AMZN class of bugs).

S3 Pickle Cache (primary read)

prices/all_data.pkl.gz — gzipped pickle, ~275 MB compressed, 4,500+ symbols. Worker Lambda loads on cold start. Rebuilt on every daily scan and nightly hygiene run.

S3 Parquet Shadow (added Apr 2026)

Parallel write to prices/all_data.parquet on every scan and hygiene run. Columnar zstd compression, schema-enforced via pyarrow. Enables partial reads (one symbol without loading the full blob) and DuckDB SQL queries. Staged migration path: parquet will replace pickle post-AL2023 Lambda upgrade.

In-Memory Cache

ScannerService.data_cache: Dict[str, DataFrame]. 1.5-2 GB footprint on Worker Lambda (4 GB configured).

CDN Pre-computation

Dashboard JSON exported to S3 daily. CloudFront serves with 1-hour TTL for zero-latency dashboard loads.

Data Integrity Filters (Apr 2026)

Scanner + backtester enforce _is_data_quality_ok() before a symbol enters candidates: rejects price/DWAP ratios > 50× or < 0.02, abrupt single-day moves > 65% (log-return > 0.5), and >10-day gaps between bars (ticker-reuse signature). Catches unadjusted splits, stitched histories, and delisting artifacts in real time.

DuckDB SQL Layer (Apr 2026)

Ad-hoc SQL queries over the S3 parquet via data_export_service.query_parquet(). Lambda-compatible (downloads parquet to /tmp due to AL2 glibc; AL2023 upgrade will enable native httpfs). Powers the parquet_diagnose handler for universe-wide corruption scans in seconds.

Layer 2 Data Hygiene — Nightly Pipeline (added Apr 15 2026)

Automated protection against silent data drift from corporate actions. Runs every weekday at 6:30 PM ET via nightly_data_hygiene EventBridge rule:

  1. Asset-ID verification — compares each symbol's stored Alpaca asset UUID to the current value. Any change signals ticker reuse → auto-quarantine + admin alert.
  2. Corp-actions poll — queries Alpaca corporate-actions API for splits, dividends, spinoffs, mergers, delistings in the last 36h. 600+ events typical per day.
  3. Split auto-refetch — for each detected split, fetches clean split-adjusted history with Adjustment.SPLIT, overwrites in cache.
  4. Dual-store re-export — updated cache written to both S3 pickle AND parquet (keeps stores in lockstep).
  5. Post-actions diagnose — DuckDB corruption scan over the fresh parquet; flags anomalies above thresholds.
  6. Admin digest email — tiered status (Healthy vs Attention Needed). Auto-remediated items (splits fixed, new symbols seen) appear as INFO; only true anomalies (ticker reuse, threshold breaches) escalate to Attention Needed.

Supporting schema: symbol_metadata (one row per symbol with asset_id + status: active/inactive/quarantined) and symbol_metadata_events (audit log of every corp action detected, with handler result).

Scanner consumes the quarantine list via symbol_metadata_service.get_excluded_symbols() at the start of each scan — quarantined symbols are silently excluded from signal generation with zero latency impact.

10. CI/CD Pipeline

Fully automated deployment via GitHub Actions on push to main. Zero-downtime updates via Lambda container image swaps.

# Trigger: push to main OR pull request to main

Stage 1: Test
  Backend: Python 3.11 → pip install → pytest -v
  Frontend: Node 18 → npm ci → npm run lint → npm test

Stage 2: Deploy (only on main push)
  1. npm run build → S3 sync → CloudFront invalidation
  2. Pre-deploy: export price data to S3
  3. Docker build (linux/amd64, --provenance=false) → ECR push
  4. Lambda update-function-code → API + Worker (same image)
  5. Post-deploy: /api/warmup (verify cold start healthy)

Critical Lambda Build Requirement: Docker BuildKit creates multi-platform manifests with attestations that Lambda rejects. All builds use --provenance=false --sbom=false flags for compatibility.
